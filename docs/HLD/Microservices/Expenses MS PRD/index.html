  • Saga Pattern: Since the expense service interacts with accounts and inventory, the choreography pattern is preferable to avoid a central orchestrator bottleneck.

4Ô∏è‚É£ Microservice Interactions‚Äã

  1. Account Validation (Pre-Expense Creation)

    • Calls GET /accounts/{accountNumber} of Accounts MS.
    • Validates that the account is active (not closed).
    • If it's a transfer, ensures the source and destination accounts exist.

  2. Inventory Update (If Purchase Expense)

    • Calls POST /inventory in Inventory MS with product details.
    • If Inventory MS is down, uses a Kafka retry mechanism.

  3. Metadata API Call

    • Calls GET /metadata/tags?category=expense to fetch applicable transaction tags.
    • Implements circuit breaker via Resilience4j.

üöÄ Overview‚Äã

This microservice is responsible for managing expenses:

  • Records single-legged (normal expense) and double-legged (fund transfer) expenses.
  • Validates accounts by calling the Accounts Microservice.
  • Publishes events to Kafka for downstream services like Inventory Microservice.
  • Accepts optional product metadata (e.g., item purchased).
  • Uses Redis for caching.
  • Future Scope: Cloud file storage for invoices/receipts.

üß† Business Logic Summary‚Äã

  • When an expense is created:

    • Validate sourceAccount is ACTIVE via Accounts MS.
    • If targetAccount exists (double-legged), validate it too.
    • Optionally attach product metadata and publish to Kafka.
    • Store expense in PostgreSQL.

  • Metadata API called to enrich tags for the expense with circuit breaker logic.
