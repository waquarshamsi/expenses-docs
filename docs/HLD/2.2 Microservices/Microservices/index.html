Microservices

Absolutely! Here's a complete list of all microservices involved in your project so far, along with a short description of each and how they interact:

🧱 Microservices in the Expense Tracker System​

1. ✅ User Microservice​

  • Purpose: Manage user profiles and authentication.

  • Responsibilities:

    • User registration and login
    • Stores user metadata
    • Issues JWT tokens

  • Interaction:

    • Used by other services for validating userId

2. ✅ Accounts Microservice​

  • Purpose: Manage various account types for users (e.g., savings, credit card, loan, investment).

  • Tech: Spring Boot MVC, PostgreSQL

  • Responsibilities:

    • CRUD for accounts
    • Account type and financial details
    • Publishes Kafka events (ACCOUNT_CREATED, etc.)
    • Caches metadata using Redis

  • Interaction:

    • Called by Expenses MS to validate account status
    • Publishes events consumed by Budgets MS

3. ✅ Expenses Microservice​

  • Purpose: Record and manage user expenses (including transfers).

  • Tech: Reactive WebFlux, PostgreSQL

  • Responsibilities:

    • Single or double-legged expense creation
    • Product metadata support
    • Publishes Kafka events to Inventory MS
    • Calls Metadata API for tags (with circuit breaker)

  • Interaction:

    • Calls Accounts MS to validate account status
    • Publishes to Kafka (e.g., EXPENSE_CREATED)
    • Sends product info to Inventory MS

4. ✅ Inventory Microservice​

  • Purpose: Track products purchased as part of expenses (e.g., electronics, groceries).

  • Tech: WebFlux, MongoDB, Kafka, Redis

  • Responsibilities:

    • Stores and retrieves product info
    • Updates inventory on product-related expenses
    • Caches frequently queried products

  • Interaction:

    • Listens to Kafka events from Expenses MS
    • Optionally provides inventory dashboard via GraphQL

5. ✅ Budgets Microservice​

  • Purpose: Manage budgets for different categories/accounts.

  • Responsibilities:

    • Budget creation and tracking
    • Analyze expenses and provide feedback
    • React to account/expense changes

  • Interaction:

    • Consumes Kafka events from Accounts and Expenses
    • May call Expenses MS to calculate category-wise spend

6. ✅ Metadata Microservice​

  • Purpose: Provide dynamic tags, categories, institutions, or other reference data.

  • Tech: REST/gRPC

  • Responsibilities:

    • Provides standard metadata like tags for expenses
    • Can be cached or wrapped with circuit breaker

  • Interaction:

    • Called by Expenses MS when user attaches tags to an expense

7. ✅ Orchestrator Saga Service (optional)​

  • Purpose: Handle coordination logic for complex multi-step workflows (if choreography becomes unmanageable).
  • Note: You’ve opted for choreography-based saga, so this is currently not used, but could be added later if needed.
